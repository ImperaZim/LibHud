# 2.0

Released April 2025

**For PocketMine-MP 5.42.X**

LibHud 2.0 is a complete standalone extraction of the HUD subsystem previously embedded in EasyLibrary's `imperazim\components\hud` namespace. The entire HUD API has been relocated to `imperazim\hud\*`, decoupled from the monolith's `PluginComponent` lifecycle, and reorganized into focused sub-namespaces: `bossbar`, `scoreboard`, `message`, `nametag`, `cooldown`, `notification`, `composer`, and `tablist`. Every class is now self-contained, uses static manager patterns or plain composition, and requires no dependency on EasyLibrary or any other LibHud module. This release also introduces several new classes (AnimatedBossBar, BossBarPool, CooldownHUD, NotificationQueue, HudComposer, TitleManager, ActionBarManager, TabList, ScoreBoardTemplates, PeriodicUpdate) that did not exist in the original EasyLibrary component.

---

## Breaking Changes

### Namespace migration

- **Changed** all classes from `imperazim\components\hud\*` to `imperazim\hud\*`. This is the fundamental breaking change of v2.0. The old namespace no longer exists in this library. Plugins that consumed the embedded EasyLibrary HUD classes must update all `use` imports.
- **Why**: Standalone extraction requires its own root namespace. The `components\hud` prefix was an EasyLibrary internal convention that coupled HUD code to the monolith's directory structure.

### Removed EasyLibrary coupling

- **Removed** dependency on `PluginComponent` base class. All HUD classes previously extended or were managed by `PluginComponent`, which tied them to EasyLibrary's plugin lifecycle. Now each manager class is a plain `final class` with static methods, and `LibHud` is a lightweight `PluginBase` that only registers event listeners.
- **Removed** the `BossBarManager` integration layer. That class remains in EasyLibrary's `components\hud\` as a bridge for plugins that still use the monolith. LibHud 2.0 replaces it with `BossBarPool` which provides conflict-free multi-bar management.
- **Why**: Standalone libraries must not depend on the host monolith. Composition over inheritance: static managers are simpler to use from any plugin without requiring a specific base class.

### Error handling

- **Removed** `new \crashdump($e)` calls that existed in the original EasyLibrary embedded code. All exception handling now uses `GlobalLogger::get()->logException($e)` for proper PocketMine-MP logging, or throws domain-specific `HudException`.
- **Why**: `\crashdump` was an EasyLibrary-internal utility. A standalone library must use standard PocketMine logging facilities.

## Core

### `imperazim\hud\LibHud`

The main plugin entry point. A minimal `PluginBase` that registers itself as a `Listener` and handles two responsibilities: validating incoming BossBar packets and cleaning up all HUD caches when players disconnect.

#### Highlights

- **Added** `LibHud` as a `final class` extending `PluginBase` and implementing `Listener`. This replaces the previous pattern where HUD cleanup was scattered across EasyLibrary's main plugin class.
- **Added** `onDataPacketReceiveEvent(DataPacketReceiveEvent)` handler that validates `BossEventPacket` types. Register/unregister packets from the client are logged; all other types are cancelled with a warning. This prevents clients from spoofing boss bar state.
- **Added** `onPlayerQuit(PlayerQuitEvent)` handler that calls `cleanup()` on all six static managers: `TabList`, `ScoreBoardManager`, `NameTagManager`, `BossBarPool`, `CooldownHUD`, and `NotificationQueue`. This is centralized cleanup that ensures zero memory leaks.
- **Why**: A single entry point for event handling keeps the library self-managing. Plugins that use LibHud do not need to remember to call cleanup on each manager individually.

#### Usage

```php
<?php
// plugin.yml loads LibHud at STARTUP automatically.
// No manual initialization needed for basic HUD features.
// For CooldownHUD and NotificationQueue, call init() in your plugin:

use imperazim\hud\cooldown\CooldownHUD;
use imperazim\hud\notification\NotificationQueue;

// In your plugin's onEnable():
CooldownHUD::init($this);
NotificationQueue::init($this);
```

## Exception

### `imperazim\hud\exception\HudException`

#### Highlights

- **Added** `HudException` as a `final class` extending `RuntimeException`. This is the single domain exception for all HUD-related errors.
- **Changed** from generic `\Exception` or `PluginException` (used in EasyLibrary) to a dedicated exception class. This allows plugins to catch HUD errors specifically without catching unrelated exceptions.
- **Why**: Domain-specific exceptions are a best practice for library design. They make error handling predictable and allow callers to distinguish HUD failures from other runtime errors.

#### Usage

```php
<?php

use imperazim\hud\exception\HudException;
use imperazim\hud\bossbar\BossBar;

try {
    $bar = new BossBar();
    $bar->setEntity($closedEntity); // throws HudException
} catch (HudException $e) {
    $logger->warning("HUD error: " . $e->getMessage());
}
```

## BossBar

### `imperazim\hud\bossbar\BossBar`

The core boss bar implementation. Spawns an invisible entity on the client and uses `BossEventPacket` to render a bar with title, subtitle, percentage, and color.

#### Highlights

- **Added** fluent API -- all setters return `static` for method chaining (`$bar->setTitle("...")->setPercentage(0.5)->setColor(BossBarColor::GREEN)`).
- **Added** `getFullTitle(): string` that combines title and subtitle with a double newline separator, encoded as UTF-8.
- **Added** `hideFrom(Player[])` and `hideFromAll()` to temporarily hide the bar without removing players from the internal list.
- **Added** `showTo(Player[])` and `showToAll()` to re-show a hidden bar.
- **Added** `setEntity(?Entity)` to attach the boss bar to a real entity (e.g., a boss mob). Validates that the entity is not closed or flagged for despawn; throws `HudException` if invalid.
- **Added** `resetEntity(bool $removeEntity)` to detach from a real entity and return to the fake invisible slime.
- **Added** `getAttributeMap(): AttributeMap` as a public accessor for the bar's health attribute.
- **Changed** all packet-sending methods are wrapped in try/catch with `GlobalLogger::get()->logException()` instead of crashing the server. This makes boss bars resilient to transient network errors.
- **Changed** the fake entity type from a custom implementation to `minecraft:slime` with zero scale, zero bounding box, and `SILENT | INVISIBLE | NO_AI | FIRE_IMMUNE` flags. This ensures the entity is completely invisible and non-interactive on all Bedrock clients.
- **Removed** dependency on `PluginComponent`. `BossBar` is now a plain class that can be instantiated anywhere.

#### Usage

```php
<?php

use imperazim\hud\bossbar\BossBar;
use pocketmine\network\mcpe\protocol\types\BossBarColor;

// Create and configure
$bar = new BossBar();
$bar->setTitle("World Boss: Dragon")
    ->setSubTitle("Phase 1")
    ->setPercentage(1.0)
    ->setColor(BossBarColor::RED);

// Show to a player
$bar->addPlayer($player);

// Update progress
$bar->setPercentage(0.5);

// Temporarily hide/show
$bar->hideFromAll();
$bar->showToAll();

// Attach to a real entity
$bar->setEntity($dragonEntity);

// Remove a specific player
$bar->removePlayer($player);

// Remove all players
$bar->removeAllPlayers();
```

### `imperazim\hud\bossbar\AnimatedBossBar`

A boss bar with built-in animated progress. Extends `BossBar` and uses PocketMine's scheduler to interpolate the percentage between two values over a specified duration in ticks.

#### Highlights

- **Added** `AnimatedBossBar` as a new class (did not exist in EasyLibrary). Extends `BossBar` and requires a `Plugin` instance for task scheduling.
- **Added** `startAnimation(int $duration, float $from, float $to, ?Closure $onComplete)` that linearly interpolates the bar percentage from `$from` to `$to` over `$duration` ticks (20 ticks = 1 second). An optional callback fires when the animation finishes.
- **Added** `startLoop(int $cycleDuration, float $from, float $to)` for ping-pong animation that continuously bounces between `$from` and `$to`. Useful for loading indicators or ambient effects.
- **Added** `stopAnimation()` to cancel a running animation.
- **Added** `isAnimating(): bool` to check if an animation is currently active.
- **Added** `getAnimationProgress(): float` to read the current progress (0.0 - 1.0) of the animation.
- **Why**: Animated boss bars are a common pattern for loading screens, countdowns, and boss fight phases. Previously, plugins had to manage their own scheduled tasks to animate a bar. This class encapsulates that pattern.

#### Usage

```php
<?php

use imperazim\hud\bossbar\AnimatedBossBar;
use pocketmine\network\mcpe\protocol\types\BossBarColor;

$bar = new AnimatedBossBar($plugin);
$bar->setTitle("Loading Map...")
    ->setColor(BossBarColor::YELLOW)
    ->addPlayer($player);

// Animate from 0% to 100% over 5 seconds (100 ticks)
$bar->startAnimation(
    duration: 100,
    from: 0.0,
    to: 1.0,
    onComplete: function (AnimatedBossBar $bar): void {
        $bar->setTitle("Map Loaded!");
        // Auto-remove after 2 seconds
    }
);

// Or use a looping ping-pong animation
$bar->startLoop(cycleDuration: 60, from: 0.2, to: 0.8);

// Check state
if ($bar->isAnimating()) {
    $progress = $bar->getAnimationProgress();
}

// Stop manually
$bar->stopAnimation();
```

### `imperazim\hud\bossbar\DiverseBossBar`

A boss bar that supports per-player customization. Each player can see a different title, subtitle, percentage, and color on the same bar instance.

#### Highlights

- **Added** per-player attribute maps -- each player gets their own `AttributeMap` clone when added, so health percentages are independent.
- **Added** `setTitleFor(Player[], string)` and `getTitleFor(Player): string` to override the title for specific players while keeping a shared default.
- **Added** `setSubTitleFor(Player[], string)` and `getSubTitleFor(Player): string` for per-player subtitles.
- **Added** `setPercentageFor(Player[], float)` and `getPercentageFor(Player): float` for per-player progress.
- **Added** `setColorFor(Player[], int)` and `getColorFor(Player): int` for per-player bar colors.
- **Added** `getFullTitleFor(Player): string` that resolves the per-player title + subtitle combination, falling back to the shared defaults.
- **Added** `resetFor(Player)` and `resetForAll()` to clear per-player overrides and revert to defaults.
- **Added** `getPlayerAttributeMap(Player): AttributeMap` for direct access to a player's attribute map.
- **Added** `getPropertyManager(?Player): EntityMetadataCollection` overrides the parent to return per-player metadata when a player is provided.
- **Changed** `sendBossPacket`, `sendBossTextPacket`, `sendBossHealthPacket` are all overridden to send per-player data instead of shared data.
- **Added** `sendAttributesPacket(Player[])` for syncing per-player health attributes.
- **Why**: In mini-game servers, different teams or roles need to see different boss bar states. Previously, plugins had to create separate `BossBar` instances per configuration. `DiverseBossBar` consolidates this into a single bar with per-player overrides, reducing memory usage and simplifying management.

#### Usage

```php
<?php

use imperazim\hud\bossbar\DiverseBossBar;
use pocketmine\network\mcpe\protocol\types\BossBarColor;

$bar = new DiverseBossBar();
$bar->setTitle("Default Title")
    ->setPercentage(1.0)
    ->setColor(BossBarColor::PURPLE);

// Add players
$bar->addPlayer($playerA);
$bar->addPlayer($playerB);

// Per-player customization
$bar->setTitleFor([$playerA], "Team Red: 5 kills");
$bar->setTitleFor([$playerB], "Team Blue: 3 kills");
$bar->setPercentageFor([$playerA], 0.75);
$bar->setPercentageFor([$playerB], 0.45);
$bar->setColorFor([$playerA], BossBarColor::RED);
$bar->setColorFor([$playerB], BossBarColor::BLUE);

// Read per-player state
$titleA = $bar->getTitleFor($playerA); // "Team Red: 5 kills"
$pctB = $bar->getPercentageFor($playerB); // 0.45

// Reset a player to defaults
$bar->resetFor($playerA);
```

### `imperazim\hud\bossbar\BossBarPool`

A static registry that manages multiple named boss bars and enforces the Bedrock client limit of 5 simultaneous bars per player.

#### Highlights

- **Added** `BossBarPool` as a new class (did not exist in EasyLibrary). Replaces the old `BossBarManager` that was part of EasyLibrary's `PluginComponent` system.
- **Added** `MAX_BARS_PER_PLAYER = 5` constant reflecting the Bedrock client hard limit.
- **Added** `create(string $id): BossBar` that creates and registers a bar. Returns the existing bar if one with the same id already exists (idempotent).
- **Added** `get(string $id): ?BossBar` for retrieving a bar by id.
- **Added** `show(Player, string $id): bool` that adds a player to a bar. Returns `false` if the bar does not exist or the player has reached the 5-bar limit.
- **Added** `hide(Player, string $id): void` that removes a player from a bar.
- **Added** `remove(string $id): void` that despawns a bar from all players and unregisters it.
- **Added** `getActive(Player): string[]` to list all active bar ids for a player.
- **Added** `getActiveCount(Player): int` to get the count of active bars.
- **Added** `exists(string $id): bool` to check if a bar is registered.
- **Added** `cleanup(Player): void` for use on `PlayerQuitEvent` -- removes the player from all bars and clears tracking data.
- **Why**: Without a central registry, multiple plugins can accidentally exceed the Bedrock 5-bar limit, causing bars to silently fail. `BossBarPool` prevents this with per-player tracking and a capacity check. It also prevents id collisions by returning existing bars on duplicate `create()` calls.

#### Usage

```php
<?php

use imperazim\hud\bossbar\BossBarPool;
use pocketmine\network\mcpe\protocol\types\BossBarColor;

// Create a named bar
$bar = BossBarPool::create("quest_progress");
$bar->setTitle("Quest: Defeat 10 Mobs")
    ->setPercentage(0.5)
    ->setColor(BossBarColor::GREEN);

// Show to a player (respects 5-bar limit)
$success = BossBarPool::show($player, "quest_progress");
if (!$success) {
    // Player is at the Bedrock limit of 5 bars
}

// Check active bars
$active = BossBarPool::getActive($player); // ["quest_progress"]
$count = BossBarPool::getActiveCount($player); // 1

// Hide from a player
BossBarPool::hide($player, "quest_progress");

// Remove entirely
BossBarPool::remove("quest_progress");

// Check existence
if (BossBarPool::exists("quest_progress")) {
    $bar = BossBarPool::get("quest_progress");
}
```

## ScoreBoard

### `imperazim\hud\scoreboard\ScoreBoard`

The data model for a scoreboard. Holds the display configuration and score lines, and converts itself to PocketMine network packets.

#### Highlights

- **Added** display slot constants: `DISPLAY_SLOT_LIST`, `DISPLAY_SLOT_SIDEBAR`, `DISPLAY_SLOT_BELOW_NAME`.
- **Added** fluent setters for all properties: `setDisplaySlot()`, `setDisplayName()`, `setObjectiveName()`, `setCriteriaName()`, `setSortOrder()` -- all return `self`.
- **Added** `toPacket(): SetDisplayObjectivePacket` that converts the scoreboard configuration into a network packet.
- **Added** `getLines(): SetScorePacket` that converts all score lines into a single packet with `TYPE_CHANGE`.
- **Added** `setLine(ScoreLine): self` to append a single line, and `setLines(ScoreLine[]): self` to replace all lines at once. Both auto-set the `objectiveName` on each line.
- **Changed** from the previous EasyLibrary implementation where scoreboard creation and packet sending were interleaved. Now the `ScoreBoard` is a pure data object, and `ScoreBoardManager` handles the sending. This separation of concerns makes scoreboards testable and reusable.

#### Usage

```php
<?php

use imperazim\hud\scoreboard\ScoreBoard;
use imperazim\hud\scoreboard\ScoreLine;
use imperazim\hud\scoreboard\ScoreBoardManager;

$board = new ScoreBoard("My Server");
$board->setDisplaySlot(ScoreBoard::DISPLAY_SLOT_SIDEBAR)
      ->setSortOrder(0)
      ->setLine(new ScoreLine(1, ""))
      ->setLine(new ScoreLine(2, "Player: Steve"))
      ->setLine(new ScoreLine(3, "Kills: 12"))
      ->setLine(new ScoreLine(4, "Deaths: 3"))
      ->setLine(new ScoreLine(5, ""));

ScoreBoardManager::sendToPlayer($player, $board);
```

### `imperazim\hud\scoreboard\ScoreLine`

Represents a single line on a scoreboard. Wraps PocketMine's `ScorePacketEntry` with validation.

#### Highlights

- **Added** score validation: constructor and `setScore()` throw `InvalidArgumentException` if the score is outside 1-15. This prevents silent packet failures from out-of-range scores.
- **Added** `TYPE_FAKE_PLAYER` constant (alias for `ScorePacketEntry::TYPE_FAKE_PLAYER`), which is the standard type for text-only scoreboard lines.
- **Added** `toEntry(): ScorePacketEntry` for converting to the network format.
- **Added** `getMessage(): string`, `setMessage(string)`, `setScore(int)`, `setObjectiveName(string)`, `setScoreboardId(int)` accessors.
- **Why**: Raw `ScorePacketEntry` has no validation and requires manual field assignment. `ScoreLine` provides a validated, constructor-based API that reduces boilerplate and catches errors early.

#### Usage

```php
<?php

use imperazim\hud\scoreboard\ScoreLine;

// Create a line at position 3 with text
$line = new ScoreLine(3, "Coins: 1500");

// Update the message
$line->setMessage("Coins: 1750");

// Access values
$text = $line->getMessage(); // "Coins: 1750"
```

### `imperazim\hud\scoreboard\ScoreBoardManager`

Static manager that handles sending, removing, updating, and cleaning up scoreboards per player. Tracks active scoreboards by player name.

#### Highlights

- **Added** `sendToPlayer(Player, ScoreBoard): void` -- automatically removes any existing scoreboard before sending the new one. Sends both the `SetDisplayObjectivePacket` and the `SetScorePacket` for all lines.
- **Added** `removeFromPlayer(Player): void` -- sends a `RemoveObjectivePacket` and clears the tracked scoreboard.
- **Added** `clearLine(Player, int $score): void` -- removes a single line by score position.
- **Added** `clearAllLines(Player): void` -- removes all 15 possible lines without removing the scoreboard itself.
- **Added** `updateToPlayer(Player): void` -- re-sends the current scoreboard (useful after modifying lines).
- **Added** `getScoreBoardFromPlayer(Player): ?ScoreBoard` -- retrieves the active scoreboard for a player.
- **Added** `cleanup(Player): void` -- silently removes the player's tracked scoreboard from memory (called on quit).
- **Changed** from instance-based management in EasyLibrary to a fully static API. This means no manager instance needs to be passed around; any code can access the scoreboard state for any player.
- **Why**: A static manager simplifies the API for the most common use case -- showing one scoreboard per player. The automatic removal of the previous scoreboard prevents the common bug of overlapping objectives.

#### Usage

```php
<?php

use imperazim\hud\scoreboard\ScoreBoard;
use imperazim\hud\scoreboard\ScoreLine;
use imperazim\hud\scoreboard\ScoreBoardManager;

// Send a scoreboard
$board = new ScoreBoard("Stats");
$board->setLine(new ScoreLine(1, ""))
      ->setLine(new ScoreLine(2, "Kills: 5"))
      ->setLine(new ScoreLine(3, "Deaths: 2"))
      ->setLine(new ScoreLine(4, ""));

ScoreBoardManager::sendToPlayer($player, $board);

// Clear a single line
ScoreBoardManager::clearLine($player, 2);

// Clear all lines
ScoreBoardManager::clearAllLines($player);

// Re-send after modification
ScoreBoardManager::updateToPlayer($player);

// Get current scoreboard
$current = ScoreBoardManager::getScoreBoardFromPlayer($player);

// Remove entirely
ScoreBoardManager::removeFromPlayer($player);
```

### `imperazim\hud\scoreboard\ScoreBoardTemplates`

Pre-built scoreboard templates with dynamic placeholder resolution. Provides ready-to-use scoreboards for common scenarios without manual line construction.

#### Highlights

- **Added** `ScoreBoardTemplates` as a new class (did not exist in EasyLibrary).
- **Added** `lobby(Player, string $serverName)` template showing player name, online count, max players, and TPS.
- **Added** `gameStats(Player, string $title, array $stats)` template that accepts a label-value map and renders up to 12 stat lines.
- **Added** `playerInfo(Player, string $title)` template showing name, health, XP level, ping, and world.
- **Added** `fromTemplate(Player, string $title, string[] $lines)` that resolves placeholders in arbitrary line arrays. Supported placeholders: `{player}`, `{online}`, `{max}`, `{tps}`, `{health}`, `{max_health}`, `{level}`, `{ping}`, `{world}`, `{x}`, `{y}`, `{z}`.
- **Why**: Scoreboard setup involves repetitive boilerplate. Templates reduce this to a single method call for common patterns, and `fromTemplate()` supports custom layouts with the same placeholder system.

#### Usage

```php
<?php

use imperazim\hud\scoreboard\ScoreBoardTemplates;

// Quick lobby scoreboard
ScoreBoardTemplates::lobby($player, "MyServer");

ScoreBoardTemplates::gameStats($player, "BedWars", [
    "Kills" => "5",
    "Deaths" => "2",
    "Beds Broken" => "1",
    "Final Kills" => "3",
]);

// Player info panel
ScoreBoardTemplates::playerInfo($player, "Profile");

// Custom template with placeholders
ScoreBoardTemplates::fromTemplate($player, "{player}'s HUD", [
    "",
    "Online: {online}/{max}",
    "TPS: {tps}",
    "Health: {health}/{max_health}",
    "World: {world}",
    "Pos: {x}, {y}, {z}",
    "",
]);
```

### `imperazim\hud\scoreboard\PeriodicUpdate`

Automatically refreshes scoreboards on a configurable timer using factory closures. Handles player disconnection gracefully.

#### Highlights

- **Added** `PeriodicUpdate` as a new class (did not exist in EasyLibrary).
- **Added** constructor `__construct(Plugin $plugin, int $intervalTicks = 20)` that immediately starts the update loop.
- **Added** `register(Player, string $key, Closure $factory)` where `$factory` is `fn(Player): ScoreBoard`. The factory is called every tick interval, and its return value is sent to the player via `ScoreBoardManager`.
- **Added** `unregister(Player, string $key)` and `unregisterAll(Player)` to stop updates for specific keys or all keys.
- **Added** `has(Player, ?string $key): bool` to check if a player has registered updaters.
- **Added** `start()` and `stop()` to control the scheduler task.
- **Added** `setInterval(int $ticks)` to change the update frequency at runtime (restarts the task).
- **Added** `getPlayerCount(): int` to see how many players have active updaters.
- **Added** automatic cleanup of disconnected players during each tick cycle.
- **Why**: Dynamic scoreboards (showing ping, TPS, health, etc.) require periodic updates. Previously, each plugin created its own repeating task. `PeriodicUpdate` centralizes this into a single scheduled task with keyed factories, reducing scheduler overhead and preventing duplicate update tasks.

#### Usage

```php
<?php

use imperazim\hud\scoreboard\PeriodicUpdate;
use imperazim\hud\scoreboard\ScoreBoard;
use imperazim\hud\scoreboard\ScoreLine;
use pocketmine\Server;
use pocketmine\player\Player;

// Create an updater that refreshes every second (20 ticks)
$updater = new PeriodicUpdate($plugin, intervalTicks: 20);

// Register a dynamic scoreboard for a player
$updater->register($player, "live_stats", function (Player $p): ScoreBoard {
    $sb = new ScoreBoard("Live Stats");
    $sb->setLine(new ScoreLine(1, ""))
       ->setLine(new ScoreLine(2, "Online: " . count(Server::getInstance()->getOnlinePlayers())))
       ->setLine(new ScoreLine(3, "TPS: " . Server::getInstance()->getTicksPerSecond()))
       ->setLine(new ScoreLine(4, "Ping: " . $p->getNetworkSession()->getPing() . "ms"))
       ->setLine(new ScoreLine(5, ""));
    return $sb;
});

// Check if registered
if ($updater->has($player, "live_stats")) {
    // ...
}

// Change interval to 2 seconds
$updater->setInterval(40);

// Unregister a specific updater
$updater->unregister($player, "live_stats");

// Stop the entire update loop
$updater->stop();
```

## Message

### `imperazim\hud\message\TitleManager`

Static API for sending title and subtitle screen messages with configurable fade timing.

#### Highlights

- **Added** `TitleManager` as a new class (did not exist in EasyLibrary).
- **Added** `send(Player, string $title, string $subtitle, int $fadeIn, int $stay, int $fadeOut)` that sends timing, subtitle (before title, as required by Bedrock protocol), and title packets in the correct order.
- **Added** `sendSubtitle(Player, string $subtitle)` shorthand that sends an empty title with the subtitle.
- **Added** `sendToAll(Player[], string $title, string $subtitle, int $fadeIn, int $stay, int $fadeOut)` for broadcasting.
- **Added** `clear(Player)` to immediately dismiss the current title display.
- **Added** `reset(Player)` to reset all title settings (timing, text) to defaults.
- **Changed** from PocketMine's built-in `Player::sendTitle()` which does not expose all timing controls. `TitleManager` sends raw `SetTitlePacket` packets for full control.
- **Why**: The built-in `Player::sendTitle()` has limited timing control and does not support clearing/resetting independently. `TitleManager` provides the complete Bedrock title API surface.

#### Usage

```php
<?php

use imperazim\hud\message\TitleManager;

// Full title with timing (ticks)
TitleManager::send(
    $player,
    title: "Welcome!",
    subtitle: "Enjoy your stay",
    fadeIn: 10,
    stay: 40,
    fadeOut: 10
);

// Subtitle only
TitleManager::sendSubtitle($player, "Watch out!");

// Broadcast to all players
TitleManager::sendToAll(
    $server->getOnlinePlayers(),
    "Server Announcement",
    "Restarting in 5 minutes"
);

// Clear or reset
TitleManager::clear($player);
TitleManager::reset($player);
```

- **Why**: Toast notifications are visually distinct from other HUD elements and are useful for quest completions, achievements, and alerts. The raw `ToastRequestPacket` is simple but easy to forget; this wrapper provides a discoverable API.

ToastManager::send($player, "Quest Complete!", "You earned 50 coins");

// Broadcast to all
ToastManager::sendToAll(
    $server->getOnlinePlayers(),
    "Server",
    "Restarting in 5 minutes"
);

## NameTag

### `imperazim\hud\nametag\NameTagManager`

Static manager for custom nametags above players with placeholder support and per-observer overrides.

#### Highlights

- **Added** `NameTagManager` as a new class (did not exist in EasyLibrary).
- **Added** `set(Player $target, string $format)` for global custom nametags visible to all online players. Supports placeholders: `{name}`, `{health}`, `{max_health}`, `{level}`.
- **Added** `setFor(Player $observer, Player $target, string $format)` for per-observer nametags. Only the specified observer sees the custom format; other players see the global format or default.
- **Added** `reset(Player $target)` to revert a player's nametag to their original `Player::getNameTag()` value.
- **Added** `refresh(Player $target)` to re-send the nametag to all online observers. Useful after the target's health/level changes.
- **Added** `cleanup(Player)` for use on `PlayerQuitEvent`. Removes the player as both a target and an observer from all tracking maps.
- **Added** per-observer cleanup that iterates all targets and removes the disconnecting player as an observer, preventing stale references.
- **Why**: Custom nametags are essential for rank prefixes, team colors, and health displays. PocketMine's `Player::setNameTag()` is global and does not support per-observer overrides. `NameTagManager` uses `SetActorDataPacket` to send different nametags to different observers, enabling team-based visibility and role indicators.

#### Usage

```php
<?php

use imperazim\hud\nametag\NameTagManager;

// Set a global custom nametag (visible to all)
NameTagManager::set($player, "[VIP] {name} {health}HP");

// Set a per-observer nametag (only $observer sees this)
NameTagManager::setFor($observer, $target, "[Enemy] {name}");

// Refresh after health change
NameTagManager::refresh($player);

// Reset to default
NameTagManager::reset($player);
```

## Cooldown

### `imperazim\hud\cooldown\CooldownHUD`

Visual cooldown indicator using a temporary boss bar that drains over time and auto-removes when the cooldown expires.

#### Highlights

- **Added** `CooldownHUD` as a new class (did not exist in EasyLibrary).
- **Added** `init(Plugin)` for providing a scheduler. Falls back to scanning for any enabled plugin if not initialized (with a `RuntimeException` if none are available).
- **Added** `show(Player, string $key, string $title, float $seconds, int $color, int $updateIntervalTicks)` that creates a `BossBar`, sets it to 100%, and schedules a repeating task that decreases the percentage based on real elapsed time (`microtime(true)`). The bar auto-removes when the cooldown expires or the player disconnects.
- **Added** `hide(Player, string $key)` to manually cancel a cooldown bar before it expires.
- **Added** `isActive(Player, string $key): bool` to check if a cooldown is currently displayed.
- **Added** `cleanup(Player)` for use on `PlayerQuitEvent`. Finds all bars prefixed with the player's name and cancels them.
- **Why**: Ability cooldowns are a universal game mechanic, and a visual indicator greatly improves UX. Previously, plugins had to create a boss bar, schedule a task, track the elapsed time, update the percentage, and clean up on disconnect -- all manually. `CooldownHUD` encapsulates this entire pattern in a single `show()` call.

#### Usage

```php
<?php

use imperazim\hud\cooldown\CooldownHUD;
use pocketmine\network\mcpe\protocol\types\BossBarColor;

// Initialize once in your plugin's onEnable()
CooldownHUD::init($plugin);

// Show a 5-second cooldown bar
CooldownHUD::show(
    $player,
    key: "ability.dash",
    title: "Dash Cooldown",
    seconds: 5.0,
    color: BossBarColor::RED
);

// Show a 10-second cooldown with custom update rate
CooldownHUD::show(
    $player,
    key: "item.potion",
    title: "Potion Cooldown",
    seconds: 10.0,
    color: BossBarColor::GREEN,
    updateIntervalTicks: 4 // Update every 0.2s instead of 0.1s
);

// Check if active
if (CooldownHUD::isActive($player, "ability.dash")) {
    // Prevent re-use
}

// Cancel early
CooldownHUD::hide($player, "ability.dash");
```

## Notification

### `imperazim\hud\notification\NotificationQueue`

Priority queue that serializes toast, title, and action bar notifications so they display one at a time without overlap.

#### Highlights

- **Added** `NotificationQueue` as a new class (did not exist in EasyLibrary).
- **Added** `init(Plugin)` for providing a scheduler. Falls back to scanning for any enabled plugin.
- **Added** `toast(Player, string $title, string $body, int $priority, int $durationTicks)` to enqueue a toast notification.
- **Added** `title(Player, string $title, string $subtitle, int $priority, int $durationTicks)` to enqueue a title notification.
- **Added** `actionBar(Player, string $message, int $priority, int $durationTicks)` to enqueue an action bar notification.
- **Added** priority-based sorting: higher priority notifications are shown first regardless of enqueue order.
- **Added** automatic sequential processing: the queue waits for the configured `durationTicks` before showing the next notification, preventing visual overlap.
- **Added** `clear(Player)` to discard all pending notifications.
- **Added** `pending(Player): int` to check the queue size.
- **Added** `cleanup(Player)` for use on `PlayerQuitEvent`.
- **Why**: When multiple systems (quests, achievements, chat, events) all send notifications simultaneously, they overlap and become unreadable. `NotificationQueue` serializes them with configurable duration and priority, ensuring each notification gets its dedicated screen time.

#### Usage

```php
<?php

use imperazim\hud\notification\NotificationQueue;

// Initialize once
NotificationQueue::init($plugin);

// Queue a toast (shown first due to higher priority)
NotificationQueue::toast(
    $player,
    title: "Quest Complete!",
    body: "+50 coins",
    priority: 10,
    durationTicks: 60 // Show for 3 seconds before next
);

// Queue a title (shown after the toast)
NotificationQueue::title(
    $player,
    title: "Level Up!",
    subtitle: "You are now level 5",
    priority: 5,
    durationTicks: 60
);

// Queue an action bar (shown last, lowest priority)
NotificationQueue::actionBar(
    $player,
    message: "Health fully restored!",
    priority: 0,
    durationTicks: 40
);

// Check pending count
$count = NotificationQueue::pending($player); // 2 (if first is already showing)

// Clear all pending
NotificationQueue::clear($player);
```

## Composer

### `imperazim\hud\composer\HudComposer`

Layered HUD composition system that manages multiple HUD elements per player with priority ordering and cleanup callbacks.

#### Highlights

- **Added** `HudComposer` as a new class (did not exist in EasyLibrary).
- **Added** `setLayer(Player, string $key, Closure $apply, ?Closure $remove, int $priority)` where `$apply` is called during `refresh()` and `$remove` is called when the layer is removed. Layers are identified by string keys and sorted by priority (lower = first).
- **Added** `removeLayer(Player, string $key)` that invokes the layer's `$remove` callback (if provided) and deletes the layer.
- **Added** `hasLayer(Player, string $key): bool` to check layer existence.
- **Added** `getLayers(Player): string[]` to list all layer keys sorted by priority.
- **Added** `refresh(Player)` that re-applies all layers in priority order. Useful after state changes to ensure all HUD elements are consistent.
- **Added** `clearAll(Player)` that invokes all `$remove` callbacks and deletes all layers for a player.
- **Added** `cleanup(Player)` for silent cleanup on disconnect (no callbacks invoked, just memory freed).
- **Added** `clearEverything()` that clears all layers for all players with remove callbacks.
- **Why**: Complex servers run multiple HUD systems simultaneously (scoreboard + boss bar + action bar + nametag). Without a composition layer, these systems can conflict or overwrite each other. `HudComposer` provides a priority-based layer system where each HUD element registers its apply/remove logic, and `refresh()` re-applies them in order. This is the "composition over inheritance" principle applied to HUD management.

#### Usage

```php
<?php

use imperazim\hud\composer\HudComposer;
use imperazim\hud\scoreboard\ScoreBoardManager;
use imperazim\hud\bossbar\BossBar;
use imperazim\hud\scoreboard\ScoreBoard;
use imperazim\hud\scoreboard\ScoreLine;

$composer = new HudComposer();

// Register a scoreboard layer
$sb = new ScoreBoard("Stats");
$sb->setLine(new ScoreLine(1, "Kills: 5"));

$composer->setLayer(
    $player,
    key: 'scoreboard',
    apply: fn() => ScoreBoardManager::sendToPlayer($player, $sb),
    remove: fn() => ScoreBoardManager::removeFromPlayer($player),
    priority: 0
);

// Register a boss bar layer
$bar = new BossBar();
$bar->setTitle("World Boss");

$composer->setLayer(
    $player,
    key: 'bossbar',
    apply: fn() => $bar->addPlayer($player),
    remove: fn() => $bar->removePlayer($player),
    priority: 1
);

// Re-apply all layers in order
$composer->refresh($player);

// Check layers
$layers = $composer->getLayers($player); // ['scoreboard', 'bossbar']
$has = $composer->hasLayer($player, 'bossbar'); // true

// Remove a specific layer (calls remove callback)
$composer->removeLayer($player, 'bossbar');

// Clear everything for a player
$composer->clearAll($player);

// Nuclear option: clear all players
$composer->clearEverything();
```

## TabList

### `imperazim\hud\tablist\TabList`

Static manager for player list (TAB menu) header and footer text using Bedrock-specific `SetTitlePacket` subtypes.

#### Highlights

- **Added** `TabList` as a new class (did not exist in EasyLibrary).
- **Added** private constants `TYPE_HEADER = 10` and `TYPE_FOOTER = 11` for the Bedrock-specific `SetTitlePacket` subtypes that control the player list header and footer. These are undocumented in PocketMine's API.
- **Added** `send(Player, string $header, string $footer)` that sends both header and footer packets and caches the values.
- **Added** `sendToAll(Player[], string $header, string $footer)` for broadcasting.
- **Added** `clear(Player)` that sends empty strings and removes the cache entry.
- **Added** `getHeader(Player): ?string` and `getFooter(Player): ?string` to read cached values.
- **Added** `cleanup(Player)` for use on `PlayerQuitEvent`.
- **Why**: Bedrock TAB list header/footer is controlled via `SetTitlePacket` with special type values (10 and 11) that are not exposed in PocketMine's Player API. `TabList` encapsulates this protocol detail and adds caching so plugins can read the current header/footer without storing their own copies.

#### Usage

```php
<?php

use imperazim\hud\tablist\TabList;

// Set header and footer
TabList::send($player, "MyServer\n1.21.0", "play.myserver.com");

// Broadcast to all players
TabList::sendToAll(
    $server->getOnlinePlayers(),
    "Welcome!\nOnline: 42",
    "play.myserver.com"
);

// Read cached values
$header = TabList::getHeader($player); // "MyServer\n1.21.0"
$footer = TabList::getFooter($player); // "play.myserver.com"

// Clear
TabList::clear($player);
```

## Migration Guide

### From EasyLibrary `imperazim\components\hud\*` to LibHud 2.0

1. **Add LibHud** to your server's plugins directory (or require via composer: `imperazim/libhud`).

2. **Update all `use` imports**:
   ```php
   // Before (EasyLibrary embedded)
   use imperazim\components\hud\BossBar;
   use imperazim\components\hud\ScoreBoard;

   // After (LibHud 2.0)
   use imperazim\hud\bossbar\BossBar;
   use imperazim\hud\scoreboard\ScoreBoard;
   use imperazim\hud\scoreboard\ScoreBoardManager;
   ```

3. **Replace `BossBarManager`** with `BossBarPool`:
   ```php
   // Before
   BossBarManager::create($player, "quest");

   // After
   $bar = BossBarPool::create("quest");
   BossBarPool::show($player, "quest");
   ```

4. **Initialize scheduler-dependent classes** in your plugin's `onEnable()`:
   ```php
   CooldownHUD::init($this);
   NotificationQueue::init($this);
   ```

5. **Remove manual cleanup** -- LibHud handles `PlayerQuitEvent` cleanup for all static managers automatically.

6. **Remove `PluginComponent` references** -- all HUD classes are now standalone and do not extend any base class from EasyLibrary.
